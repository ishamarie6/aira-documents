# Mandatory Practice Statement

The AIRA team transformation is not a movement from human developers to uncontrolled AI automation. It is a governed maturity journey from traditional delivery to AI-native DevSecOps, where AI accelerates engineering work only within AIRA principles, AVCI, architecture fitness, policy-as-code, security controls, evidence packs, maker-checker review, and progressive autonomy boundaries. AI may recommend, draft, generate, analyze, test, and propose. AI must not silently approve, merge, deploy, bypass controls, mutate production, disable evidence, or become business authority.

# Direction Statement

Our destination is a disciplined AI-Native DevSecOps team supported by the AIRA System Builder and governed AI agents. The team will configure first, code only the business gap, verify every output, produce evidence by construction, improve continuously, and delegate only low-risk reversible actions after trust is proven. AVCI always.

# 3. Core Team Message

AI will not replace disciplined engineers. Disciplined engineers who can command, govern, verify, and improve with AI will lead the next generation of enterprise software delivery. AIRA asks the team to move beyond casual AI prompting into a governed engineering discipline where every output is secure, testable, observable, reversible, and evidence-producing from the start.

Traditional development asks: can we make it work? AI-assisted development asks: can AI help us build it faster? AIRA AI-Native DevSecOps asks: can we build it correctly, securely, repeatedly, observably, reversibly, and with evidence from the start?

# 6. Operating Principles for the Transformation
| Principle | Transformation Rule |
| --- | --- |
| Discipline first | AI use must strengthen, not bypass, engineering discipline. |
| Automation second | Automation is added only after process, ownership, evidence, and rollback are defined. |
| Intelligence third | AI reasoning is advisory until validated by policy, tests, scans, evidence, and human review. |
| AVCI always | Every output must be attributable, verifiable, classifiable, and improvable. |
| Configure first | Use MicroFunction configuration, policies, templates, contracts, rules, and reusable components before writing new code. |
| Code only the business gap | New code must be bounded, testable, adapter-based, secure, observable, reversible, and evidence-producing. |
| Fail closed | If identity, policy, guardrails, audit, evidence, model gateway, or Harness controls are unavailable, protected actions stop. |
| Human accountability | AI does not approve its own output, promote changes, accept risk, or become business authority. |

# 7. System Builder Lite to Full System Builder Operating Model

System Builder Lite begins as a governed workstation and repository workflow using ChatGPT Project guidance, VS Code, Codex, Git, Obsidian, LLM Wiki, evidence registers, AGENTS.md, CODEOWNERS, and CI/CD gates. Full System Builder becomes the governed engineering factory and change-generation control plane. Both models follow the same rule: analyze first, recommend second, generate candidates only after bounded intent is clear, validate through evidence, and promote only through human-approved change control.
| Capability | System Builder Lite | Full System Builder |
| --- | --- | --- |
| Intake | Human-curated prompt or ticket with AIRA sources. | Structured controlled intake for requirements, evidence, improvements, agents, and environments. |
| Analysis | Developer/architect guided analysis. | Impact analysis across contracts, code, data, policies, agents, evidence, and risk. |
| Generation | Branch-scoped code/config/test/document candidate. | Contract-first candidate artifacts, PR/MR-ready packages, provisioning plans, evidence packs. |
| Validation | Local tests, CI, reviewer checks, GitNexus, evidence summary. | Automated lint, policy, security, fitness, test, model, and evidence validation. |
| Promotion | Human PR/MR/CAB path. | Maker-checker, CODEOWNERS, ARB/CAB, CI/CD, registry, runtime controls. |
| Boundary | No direct production mutation. | No self-approval, no uncontrolled agents, no direct credentials, no bypass of policy or evidence. |

# 11. Observability and Runtime Toggle Rules

The team must learn to design and operate systems where traceability is not optional. Log4j2 structured logs, OpenTelemetry traces/metrics/logs, Sentry errors, Loki logs, Tempo traces, Grafana dashboards, audit events, outbox records, and evidence references must reconstruct what happened, who/what acted, under which policy, with which source, and what outcome occurred.
| Toggle Category | Allowed Runtime Behavior |
| --- | --- |
| Performance/debug telemetry | May be sampled, tuned, or temporarily reduced if approved and evidence impact is understood. |
| Security/audit/policy/classification signals | Non-disableable for protected flows. Includes authentication, authorization, denial, guardrail, model-route, secrets, DAST findings, and evidence events. |
| Business-critical MicroFunction traces | Must remain reconstructable. Sampling may not break audit chain, outbox link, policy decision, rollback, or incident investigation. |
| AI/model telemetry | Model alias, route, guardrail result, prompt template, classification, source refs, output artifact, and evidence_ref must remain captured safely. |

# 12. Auto-Heal, Auto-Learn, and Auto-Improve Governance
| Loop | Allowed Behavior | Blocked Behavior |
| --- | --- | --- |
| Auto-Heal | Detect issue, retrieve evidence, classify cause, draft remediation, run safe tests, propose PR/MR or runbook update. | Disable guardrails, scans, audit, alerts, or policy to clear a failure; mutate production without approval. |
| Auto-Learn | Convert reviewed incidents, accepted fixes, RCA, and test outcomes into candidate knowledge. | Promote uncited, stale, conflicting, or unreviewed knowledge to authoritative status. |
| Auto-Improve | Recommend refactoring, architecture fitness, performance, reliability, security, UX, test, or policy improvements. | Silently change code, configuration, prompts, policies, infrastructure, or model routes. |

# 17. Risks, Controls, and Non-Negotiable Rejection Rules
| Risk | Required Control |
| --- | --- |
