Mandatory Practice Statement

No AIRA code, configuration, MicroFunction, workflow, prompt, guardrail, policy, infrastructure, database migration, AI agent, System Builder output, or runtime configuration is ready for merge, release, activation, or promotion unless it passes approved DevSecOps gates and produces AVCI-complete evidence. DevSecOps is the governed engineering factory, not merely a pipeline script.

# 3. Purpose, Scope, Authority, and Precedence

## 3.1 Purpose

Define the mandatory AI-native DevSecOps operating model for all AIRA engineering work.

Enforce architecture, security, quality, supply-chain, evidence, and promotion gates across human-authored and AI-assisted changes.

Ensure MicroFunction-based delivery inherits the cross-cutting controls required by 10 through 10E.

Prevent uncontrolled automation, direct production mutation, direct provider calls, unmanaged agents, evidence-free promotion, and manual production drift.

Create reusable foundations for System Builder, PoC execution, CI/CD, GitNexus, evidence packs, runtime monitoring, and continuous improvement.

## 3.2 Scope
| In Scope | Out of Scope |
| --- | --- |
| Source code, tests, configuration, MicroFunctions, API/event contracts, database migrations, workflows, policies, prompts, guardrails, AI agents, IaC, devcontainers, pipelines, evidence packs, runbooks, and releases. | Direct production mutation by AI, agents, developers, scripts, notebooks, GitNexus, or unmanaged tools without approved change path. |
| Repository governance, branch protection, CODEOWNERS, CI/CD gates, SAST, SCA, DAST, SBOM, provenance, container/IaC scans, OPA/Conftest, architecture fitness, GitNexus, and promotion evidence. | Manual production DDL, unapproved emergency fixes, direct model-provider SDK calls, hardcoded secrets, unmanaged local-only changes, and shadow automation. |
| System Builder and AI-assisted development where outputs remain draft, review-ready, PR/MR-ready, or deployable only after governed approval. | AI approval of its own output, agent-held production credentials, uncontrolled tool invocation, bypass of SBAC/OPA/guardrails, or weakening of audit/evidence controls. |

## 3.3 Authority and Precedence
| Level | Authority | DevSecOps Interpretation |
| --- | --- | --- |
| L0 | Architecture Board, CAB, Security Governance, IT Head | Final authority for production-impacting release, waiver, material exception, and control relaxation. |
| L1 | 01A, 01 AVCI, 01B, Engineering Blueprint | Universal architecture, evidence, audit, traceability, and build-readiness gates. |
| L2 | AIRA-DOC-011 DevSecOps Standard v3.3 | Parent lifecycle discipline for CI/CD, security gates, evidence, supply chain, AI-assisted delivery, and promotion. |
| L3 | 20 CI/CD Guide, GitNexus, MicroFunction, API, Database, Security, Observability, System Builder Standards | Implementation controls and specialized evidence requirements. |
| L4 | Pipelines, PR/MR records, audit logs, test reports, security scans, evidence manifests, release records | Operational proof; may tighten but must not weaken governing standards. |

# 4. Governed DevSecOps Control Plane
| Principle | Mandatory Rule |
| --- | --- |
| Evidence by construction | Pipelines and tools generate evidence automatically wherever practical; evidence is not manually invented after approval. |
| Fail-closed gates | Critical security, policy, architecture, evidence, identity, guardrail, or test failures block merge or promotion unless formally waived. |
| Source truth discipline | Git is authoritative for code/config/IaC/policies/migrations; PostgreSQL is authoritative for runtime registries; Obsidian and LLM Wiki are curated projections. |
| Maker-checker | AI and humans may generate; independent checker and CODEOWNERS validate; required approvers approve. |
| Least privilege and SBAC | Humans, services, tools, and agents receive only approved skills, roles, scopes, data, and environment access. |
| Reversibility | Every material change declares rollback, safe-disable, compensation, forward-fix, deactivation, or rebuild path. |
| AVCI always | Every artifact has owner, source, evidence, classification, and improvement path. |

Runtime diagnostics may be tuned for performance by governed configuration. Mandatory audit events, security logs, OPA/SBAC decisions, classification labels, idempotency records, outbox records, critical error evidence, and evidence references are non-disableable for protected flows.

# 6. MicroFunction Cross-Cutting Capability Enforcement

The DevSecOps standard enforces the regenerated MicroFunction baseline. The following controls are mandatory for MicroFunction-related PR/MR changes, runtime configuration changes, catalog updates, Dynamic Workspace actions, AI Panel tool actions, and System Builder-generated candidates.
| Capability Area | DevSecOps Enforcement Requirement |
| --- | --- |
| DevSecOps and GitNexus | Every MicroFunction change includes CI/CD evidence, architecture fitness, security gates, GitNexus read-only impact, derived artifact summary, PR/MR AVCI summary, and rollback/safe-disable path. |
| API and eventing | OpenAPI, AsyncAPI, Kafka, Avro/JSON Schema, CloudEvents, outbox, schema evolution, idempotent consumers, DLQ, and replay are validated before integration scale. |
| Observability | Log4j2 structured logs, OpenTelemetry traces/metrics, Sentry errors, Loki logs, Tempo traces, Grafana dashboards, audit, alerting, and trace reconstruction are proven. |
| Concurrency and resilience | Idempotency, locking, retry safety, duplicate handling, heavy-load behavior, failure injection, timeouts, circuit breakers, bulkheads, compensation, and DLQ/replay are tested. |
| Auto-Heal/Learn/Improve | Issue detection, evidence retrieval, candidate fix or learning proposal, tests, security validation, rollback, and human approval are mandatory. No silent runtime mutation. |
| Security hardening | OPA/SBAC, abuse cases, authenticated DAST, secure APIs, secrets hygiene, redaction tests, and remediation evidence are present for protected paths. |
| Runtime toggles | Diagnostic logging and sampling may be adjusted by policy-approved runtime configuration. Mandatory audit, security, policy, idempotency, outbox, and critical error evidence must not be disabled. |

# 8. Security, Supply Chain, API/Event, Database, and Observability Gates

## 8.1 Security, Privacy, and Classification Gates

SAST, SCA, secret scanning, container scanning, IaC scanning, dependency review, and authenticated DAST are required based on risk and environment.

Restricted or Confidential data must not be placed in logs, traces, metric labels, prompts, screenshots, test fixtures, GitNexus reports, evidence summaries, or AI context unless explicitly approved and protected.

Secrets must be managed through approved secret stores and short-lived credentials; source code, tests, prompts, documentation, screenshots, and local .env files must not contain production secrets.

OPA/SBAC policy tests must prove deny-by-default, fail-closed, least privilege, separation of duties, and classification-aware routing.

Critical and High findings block merge and promotion unless a formal waiver records owner, risk, compensating control, expiry, and remediation evidence.

## 8.2 API/Event and Database Gates
| Area | Gate Requirement |
| --- | --- |
| OpenAPI | Contract exists before controller implementation; security schemes, idempotency headers, safe errors, classification, examples, and tests are present. |
| AsyncAPI / Kafka | Topics, producers, consumers, event headers, retry, DLQ, replay, and security posture are defined before event integration. |
| Avro/JSON Schema | Schema compatibility checks run in CI; breaking changes require ADR/TDL, consumer approval, migration, and deprecation plan. |
| CloudEvents | Event metadata carries id, source, type, specversion, time, subject where applicable, correlation, trace, classification, and evidence reference where permitted. |
| Flyway database governance | Schema, reference data, runtime config, outbox, idempotency records, indexes, RLS, and seed data are migration-controlled; manual production DDL is prohibited. |

## 8.3 Observability and Evidence Signals
| Signal | Mandatory Evidence |
| --- | --- |
| Logs | Structured Log4j2 events with trace_id, request_id, transaction_code, step_code, outcome, error_code, classification, and redaction state. |
| Traces | OpenTelemetry trace context across gateway, backend, MicroFunction runtime, database adapter, Kafka, workflow engine, and AI gateway where applicable. |
| Error monitoring | Sentry issue/release context, sanitized stack summaries, regression evidence, and incident linkage without Restricted data leakage. |
| Logs/traces storage | Loki logs and Tempo traces queryable by correlation and evidence fields for trace reconstruction. |
| Dashboards/alerts | Grafana panels and alerts for latency, error rate, policy denial, DLQ, replay, idempotency conflict, outbox lag, saturation, and Auto-* candidate metrics. |

# 9. Promotion, Runtime Toggles, RACI, and Rejection Rules

## 9.1 Promotion and Environment Governance

DEV, TEST, UAT, STAGING, PILOT, and PRODUCTION promotion paths must be separate, auditable, and policy-controlled.

Production-bound promotion requires approved change ticket, evidence pack, rollback or compensation plan, CAB/ARB approval where applicable, and post-promotion monitoring.

Manual production mutation is prohibited except approved, logged, time-bound break-glass with post-incident review and remediation evidence.

GitOps, IaC, and Flyway-controlled changes are preferred over click-ops.

Feature flags and runtime toggles must not bypass security, audit, policy, or evidence requirements.

## 9.2 Runtime Toggle Rule
| Toggle Area | Allowed Adjustment | Non-Disableable Boundary |
| --- | --- | --- |
| Logging | Debug/trace verbosity, sampling rate, diagnostic categories, and short-lived incident logging windows may be adjusted with approval. | Security logs, auth denials, policy decisions, audit events, critical errors, and evidence references. |
| Tracing | Sampling, baggage limits, and low-risk span enrichment may be tuned to manage overhead. | Trace_id propagation, critical transaction spans, policy-decision correlation, and evidence_ref. |
| Metrics | Cardinality, scrape interval, dashboard aggregation, and non-critical labels may be tuned. | SLO/SLA metrics, error rates, latency, saturation, security denial counters, outbox/DLQ metrics. |
| AI features | Advisory summaries and non-critical model-assisted diagnostics may be disabled. | Guardrails, LiteLLM route enforcement, prompt/model audit, classification checks, and human approval controls. |

## 9.3 RACI
| Role | Primary Responsibility |
| --- | --- |

| DevSecOps Lead | Owns CI/CD, evidence packs, supply-chain controls, repository rules, and promotion gates. |
| Software Development Lead | Ensures developer compliance, branch discipline, code quality, and MicroFunction implementation controls. |
| Security Architecture | Owns security gates, OPA/SBAC controls, secrets hygiene, vulnerability risk, DAST scope, and waivers. |
| QA/SDET | Owns test strategy, deterministic verification, mutation/coverage expectations, E2E readiness, and acceptance evidence. |
| Platform Engineering / SRE | Owns runners, environments, observability, reliability, runtime toggles, dashboards, alerting, and rollback readiness. |
| DBA / Data Governance | Owns Flyway governance, schema ownership, RLS, data handling, retention, and data-fix controls. |
| AI Governance / AI Engineering | Owns model routes, guardrails, agent registry, evaluations, and AI-use evidence. |
| Internal Audit | Reviews evidence completeness, traceability, exceptions, and control effectiveness. |

## 9.4 Anti-Patterns and Rejection Rules
| Anti-Pattern | Required Response |
| --- | --- |
| Pipeline passes because tests are skipped, scans are disabled, or findings are ignored. | Reject merge; require remediation or formal waiver with expiry. |
| AI-generated code introduces direct model-provider SDK calls, direct DB/Kafka calls from business logic, or controller business decisions. | Reject; refactor through ports/adapters, LiteLLM, MicroFunction envelope, and tests. |
| MicroFunction configuration executes without identity, classification, authorization, idempotency, observability, error handling, or compensation. | Reject publish; fail closed. |
| GitNexus, Obsidian, or AI summary is treated as authoritative source of code behavior. | Reject as authority; require Git/test/evidence source references. |
| Manual DDL or untracked data fix is applied outside Flyway and approval controls. | Reject and reconcile through migration governance; emergency only as break-glass. |
| Auto-Heal mutates runtime or production directly. | Block; convert to candidate PR/MR with evidence, tests, and human approval. |

Final DevSecOps Rule

A pipeline that passes functional tests but fails architecture, security, evidence, classification, observability, supply-chain, reversibility, or human-accountability gates is not AIRA-ready. It remains a draft until the required controls are proven, reviewed, and approved.
