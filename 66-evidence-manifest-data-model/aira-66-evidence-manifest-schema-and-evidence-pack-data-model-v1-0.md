Mandatory Practice Statement

AIRA evidence is not a folder of reports assembled after delivery. It is a controlled, machine-readable, source-linked, classification-aware, tamper-evident, reviewable, and improvable data model. No AIRA change is promotion-ready unless its evidence manifest can prove who made the change, what changed, what was verified, which controls passed or failed, where evidence is retained, which approvals were granted, how rollback or compensation works, and how findings feed continuous improvement.

# 1. Executive Summary

This standard defines the machine-readable evidence manifest and evidence pack data model for AIRA. It converts the principles from AVCI, DevSecOps, Policy-as-Code, Production Readiness Review, Operational Readiness Review, and the Resilience Lab into a consistent evidence schema that can be generated by CI/CD, reviewed by humans, retained for audit, and used by System Builder only as governed evidence.

The intent is to make AIRA enterprise-grade by moving from document-only governance to executable governance. Word standards define the rules. Evidence manifests prove whether the rules were followed.
| Outcome | Meaning |
| --- | --- |
| Evidence by construction | CI/CD, tests, scans, GitNexus, policy, telemetry, readiness reviews, and approvals generate evidence as part of the delivery workflow. |
| Machine-readable governance | Evidence packs are indexed by JSON/YAML manifests so automation can validate completeness, classification, lineage, retention, and review status. |
| Audit-ready traceability | Each manifest binds source change, commit, build, artifact, gate results, approvals, runtime evidence, rollback, and improvement records. |
| AI-safe evidence usage | AI agents and System Builder may summarize or recommend from evidence, but they cannot alter authoritative evidence, self-approve, or bypass policy gates. |

# 2. Purpose, Scope, and Authority

## 2.1 Purpose

Define the standard AIRA evidence manifest schema for PR/MR, release, MicroFunction, Dynamic Workspace, API/event, database, security, AI-agent, environment, resilience, and production-readiness evidence.

Standardize evidence pack folders, artifact references, gate-result structures, retention metadata, classification labels, approval evidence, waiver evidence, and improvement backlog references.

Enable CI/CD, OPA/Conftest, GitNexus, dashboards, evidence stores, and Internal Audit to validate evidence consistently.

Prevent evidence-free promotion, unverifiable AI summaries, unmanaged evidence attachments, and unclassified audit records.

## 2.2 Scope
| In Scope | Out of Scope |
| --- | --- |
| Evidence manifest JSON/YAML, evidence pack folder structure, required fields, validation rules, source and artifact references, test/scan results, GitNexus outputs, policy decisions, observability evidence, approvals, waivers, rollback, and improvement records. | Manual evidence summaries without machine-readable manifest; uncontrolled screenshots; unmanaged local-only reports; evidence stored in prompts; evidence that contains secrets or unrestricted PII; AI-generated claims without source-linked verification. |
| All AIRA change types: code, configuration, MicroFunction, API/event, database, policy, prompt, guardrail, agent, Dynamic Workspace, infrastructure, documentation, PoC, release, incident, and operational remediation. | Bypassing CI/CD, manual production mutation, manual DDL, agent self-approval, direct provider calls, GitNexus as authority, or Obsidian/LLM Wiki as authoritative source of executable truth. |

# 3. Evidence Manifest Control Plane

The evidence manifest is the index and control object for an evidence pack. The evidence pack contains retained artifacts; the manifest identifies, classifies, hashes, and links them to the source change and gate decisions.
| Control Object | Role |
| --- | --- |
| Source change | Ticket, PR/MR, commit, release candidate, environment request, incident, or AI-generated candidate. |
| Evidence manifest | Machine-readable index that binds ownership, classification, source references, gate results, artifact hashes, retention, approvals, and rollback. |
| Evidence pack | Folder or package containing reports, summaries, artifacts, telemetry links, dashboard references, approval records, and readiness evidence. |
| Evidence store | Approved retention location such as Git artifact store, OpenKM, CI artifact retention, immutable object storage, or controlled audit repository. |
| Evidence consumer | Human reviewer, CI validator, OPA policy, Internal Audit, System Builder summarizer, PRR/ORR board, CAB/ARB, or SRE readiness reviewer. |

# 5. Evidence Manifest Data Model

The evidence manifest must be small enough for CI to validate quickly and complete enough for audit reconstruction. It should reference large reports by path, URI, digest, and retention rule instead of embedding every report inline.
| Domain | Purpose | Minimum Objects |
| --- | --- | --- |
| Header | Uniquely identify the evidence pack and schema version. | manifest_id, evidence_pack_id, schema_version, document_refs, created_at, status |
| Attribution | Prove ownership and source lineage. | owner, maker, checker, approver, branch, commit_sha, ticket_id, AI_use_declaration |
| Classification | Control handling, retention, and model-route eligibility. | classification, handling_rule, retention_rule, restricted_fields, redaction_status |
| Verification | Record gate results and required evidence. | build, tests, contracts, database, policy, architecture, security, resilience |
| Artifact references | Link retained evidence without duplicating reports. | path, uri, digest, media_type, producer, generated_at, retention_store |
| Approvals and waivers | Prove human accountability and exception governance. | approval_id, role, decision, timestamp, waiver_id, expiry, compensating_control |
| Reversibility | Document rollback, safe-disable, compensation, replay, or rebuild. | rollback_ref, forward_fix_ref, compensation_ref, feature_flag_ref, recovery_test_ref |
| Improvement | Close the loop into Auto-Learn/Auto-Improve. | findings, backlog_refs, lessons, standard_update_candidates, review_date |

# 6. Required Manifest Sections and Fields
| Section | Required Fields | Blocking Condition |
| --- | --- | --- |
| manifest_header | manifest_id, schema_version, evidence_pack_id, pack_type, status, created_at, generated_by, classification | Missing manifest identity, schema version, or classification blocks evidence acceptance. |
| source_refs | ticket_id, repository, branch, commit_sha, pull_request, change_type, bounded_context, document_refs | Missing source lineage blocks promotion. |
| attribution | owner, maker, checker, approvers, CODEOWNERS, AI_tools_used, agent_ids where applicable | Ownerless or self-approved AI-generated evidence is rejected. |
| classification | data_classification, handling_rule, retention_rule, redaction_status, model_route_eligibility | Unclassified evidence or unsafe model route is rejected. |
| gate_results | build, tests, security, policy, architecture, contracts, database, supply_chain, observability, resilience | Critical failed or missing mandatory gate blocks promotion unless formally waived. |
| artifact_refs | artifact_id, category, path_or_uri, sha256, media_type, producer, generated_at, retention_store | Evidence without digest, producer, or retention location is not verifiable. |
| approval_refs | reviewer, role, decision, timestamp, signature_or_record, waiver_refs | Missing required approval blocks release. |
| reversibility | rollback_plan, safe_disable, compensation, replay_reprocess, restore_or_rebuild, tested | Missing reversible path blocks production promotion unless risk-accepted. |
| improvement | findings, limitations, backlog_items, lessons_learned, next_review | Missing improvement path weakens AVCI and must be corrected before canonical closure. |

# 7. Gate Result Model

Each gate result must be independently verifiable and policy-readable. The same structure is used whether the gate is CI build, security scan, OpenAPI diff, OPA test, GitNexus impact, PRR/ORR readiness, or Resilience Lab evidence.
| Field | Meaning | Allowed / Required Values |
| --- | --- | --- |
| gate_id | Unique gate identifier. | BUILD-001, SEC-SAST-001, POLICY-OPA-001, PRR-001, RES-DLQ-001 |
| gate_type | Type of gate. | build, test, security, policy, architecture, contract, database, observability, resilience, readiness |
| result | Decision result. | pass, fail, warning, not_applicable, waived |
| severity | Promotion effect. | advisory, warning, blocking, critical_block |
| tool | Tool or method that produced the evidence. | JUnit, Semgrep, OPA, Conftest, ZAP, Flyway, GitNexus, manual review |
| artifact_refs | Linked evidence artifacts. | One or more artifact IDs with hashes and retention locations. |
| waiver_ref | Exception record when result is waived. | Required if result = waived. |
| owner | Accountable role for result. | DevSecOps, Security, QA/SDET, DBA, SRE, Architecture, AI Governance |
| verified_at | Timestamp of verification. | ISO 8601 timestamp. |

# 9. Classification, Retention, and Redaction Rules

Evidence must be classified before storage, sharing, AI summarization, Obsidian projection, or LLM Wiki indexing.

Secrets, raw tokens, passwords, private keys, raw Restricted payloads, account numbers, raw prompts containing confidential information, and unrestricted PII must not be stored in open evidence summaries.

Evidence manifests may reference restricted evidence by digest and controlled URI without exposing the sensitive content in derived summaries.

Retention must follow classification and regulatory requirements. Expired evidence must not be deleted if it remains under audit hold, incident hold, release hold, or legal hold.

Obsidian and LLM Wiki may receive curated summaries and safe references. They are not authoritative stores for full evidence, source code, secrets, or Restricted operational payloads.
| Classification | Default Handling | AI / Indexing Rule |
| --- | --- | --- |
| PUBLIC | May be shared when approved; still source-referenced. | May be indexed if approved. |
| INTERNAL | Internal use; avoid sensitive operational details. | May be summarized with safe redaction. |
| CONFIDENTIAL | Restricted to approved teams and evidence stores. | AI use requires approved route, redaction, and source eligibility. |
| RESTRICTED | Strongest controls; minimum necessary access only. | Do not expose raw content to general AI or broad indexing; use references, hashes, and approved secure review. |

# 13. Validation, CI Enforcement, and Policy-as-Code

Evidence manifest validation shall run as part of CI/CD and as part of release-readiness review. Policy-as-Code gates may reject evidence packs that are incomplete, stale, unclassified, unsigned where required, missing hashes, missing owners, or missing required approvals.
| Validator | Checks | Blocking Example |
| --- | --- | --- |
| Schema validator | Required fields, enum values, schema version, artifact reference shape. | Missing classification or source_refs. |
| Digest validator | Each referenced artifact has hash and retained path. | Report attached without sha256 or retention location. |
| Gate validator | Required gates present for change type and risk tier. | API change missing OpenAPI diff and DAST scope. |
| Policy validator | OPA/SBAC/policy decision evidence present. | Protected action has no policy decision log. |
| Readiness validator | PRR/ORR required for production-bound release. | Production release lacks ORR runbook and dashboard evidence. |
| Resilience validator | Critical/event-driven/mutating flows have resilience evidence. | Outbox flow lacks DLQ/replay test. |
| Waiver validator | Waivers have owner, expiry, compensating control, and approval. | Expired Critical waiver used for promotion. |

Minimum CI command pattern:

python tools/aira_evidence_validate.py evidence/00-manifest/evidence-manifest.json --risk-tier production --change-type microfunction --fail-on blocking
conftest test evidence/00-manifest/evidence-manifest.json --policy policy/evidence_manifest

## 15.1 Acceptance Criteria

Every production-bound PR/MR, release, MicroFunction runtime configuration, database migration, API/event contract, AI-agent change, and environment change has an evidence manifest.

Every artifact reference includes category, path or URI, digest, classification, producer, generated timestamp, and retention store.

Gate results identify result, severity, owner, tool or method, linked artifacts, waiver if applicable, and verification timestamp.

PRR/ORR and Resilience Lab evidence can be bound to the manifest before production-like or production promotion.

CI/CD can fail closed when mandatory evidence is missing, invalid, stale, unclassified, ownerless, unverifiable, or improperly waived.

System Builder and AI agents can read safe summaries but cannot modify evidence, approve evidence, or bypass evidence validation.
