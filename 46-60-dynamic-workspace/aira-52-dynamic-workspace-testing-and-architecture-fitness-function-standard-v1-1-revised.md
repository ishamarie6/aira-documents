The frontend remains a governed renderer; backend policy, approved contracts, MicroFunctions, and authoritative data stores remain the control plane.

# 1. Executive Summary

This v1.1 revision upgrades AIRA-DOC-052 from a Dynamic Workspace testing checklist into a governed test and architecture-fitness control standard for the full Dynamic Workspace implementation set. It aligns with the revised AIRA environment, DevSecOps, CI/CD evidence, security, API, database, observability, release, audit, and Dynamic Workspace companion documents generated in the current revision sequence.

The revised standard requires tests to prove not only functional rendering but also backend-governed authorization, contract compatibility, MicroFunction binding, runtime configuration correctness, cache safety, accessibility, observability, policy-as-code behavior, AI assistant safety, idempotency, resilience, and evidence completeness.
| Strategic outcome | Required v1.1 result |
| --- | --- |
| Dynamic does not become uncontrolled | Every workspace, screen, component, widget action, admin template, runtime toggle, and AI panel capability is tested against backend policy, contracts, classification, and evidence requirements. |
| Architecture remains enforceable | CI fitness functions detect direct database calls from frontend, direct model calls, controller business logic, cross-context coupling, unregistered widgets, and MicroFunction bypass. |
| Evidence is produced by construction | Build, test, policy, security, accessibility, GitNexus, trace, audit, and evidence-pack outputs are generated by the delivery path and linked to PR/MR records. |
| Safe improvement becomes measurable | Auto-Heal, Auto-Learn, and Auto-Improve recommendations require deterministic tests, impact evidence, rollback path, and human approval before promotion. |

# 2. Purpose, Scope, and Authority

## 2.1 Purpose

Define the mandatory testing and architecture fitness functions for AIRA Dynamic Workspace, Dynamic Workspace Admin Builder, Experience Blocks, Experience Packs, AI Assistant Panel, and widget-triggered MicroFunction actions.

Convert quality expectations into executable gates, reviewable evidence, deterministic fixtures, and acceptance criteria.

Ensure frontend, backend, policy, database, event, cache, observability, and AI controls remain governed across implementation, promotion, runtime, and improvement loops.

## 2.2 Scope
| In scope | Out of scope |
| --- | --- |
| Unit, component, integration, contract, E2E, accessibility, security, policy, resilience, observability, runtime-toggle, and architecture fitness tests. | Treating visual rendering, manual screenshots, or AI-generated test claims as acceptance evidence without executable validation. |
| Dynamic Workspace APIs, screen resolution, component registry, personalization, admin templates, widget actions, MicroFunction bindings, AI panel, runtime cache, and evidence records. | Frontend-owned business authority, browser authorization decisions, direct model calls, direct database calls, uncontrolled template activation, and unmanaged runtime mutation. |
| CI/CD gates, GitNexus read-only impact evidence, derived artifacts, dashboards, audit evidence, and PR/MR AVCI summary. | Bypassing branch protection, CODEOWNERS, security gates, policy tests, human approval, or CAB/ARB controls to accelerate delivery. |

# 5. Dynamic Workspace Quality Gates
| Gate | Mandatory validation | Blocks promotion when |
| --- | --- | --- |
| DW-QG-01 Source and classification gate | Workspace, screen, component, action, evidence, and AI capability metadata have owner, version, classification, and approval state. | Missing owner, unclassified artifact, stale approval, or Restricted data routed to unauthorized surface. |
| DW-QG-02 Resolver gate | Resolver applies tenant, branch, role, skill, classification, feature flag, effective-date, and policy filtering before rendering. | Frontend receives unauthorized component, field, action, or AI capability. |
| DW-QG-03 Component registry gate | Only compiled allow-listed components render; unknown component renders safe unsupported state. | Runtime imports arbitrary components or configuration chooses unregistered code. |
| DW-QG-04 Widget action gate | Action uses generated client, idempotency key, OPA decision, MicroFunction transaction, audit, outbox, trace, and safe response envelope. | Action bypasses backend policy, MicroFunction, audit, idempotency, or contract. |
| DW-QG-05 Personalization gate | User changes remain within allowed components, mandatory widgets, classification ceiling, tenant/scope, and reversible layout rules. | Personalization creates privilege, hides mandatory compliance widgets, or breaks reset/rollback. |
| DW-QG-06 Admin Builder gate | Template authoring, checking, approval, activation, rollback, and retirement are tested with maker-checker evidence. | Admin can activate unreviewed, untested, or policy-failing templates. |
| DW-QG-07 AI panel gate | AI prompt/input/output mode, model route, retrieval eligibility, tool proposal, references, guardrails, and audit evidence are tested. | AI executes protected action, bypasses gateway, exposes unauthorized context, or lacks evidence. |

# 6. Architecture Fitness Function Catalog
| Fitness function | Enforced rule | Suggested implementation |
| --- | --- | --- |
| AFF-DW-001 Frontend is not authority | Browser code must not authorize business actions, call databases, call model providers, or own workflow decisions. | Semgrep forbidden API rules; review of generated clients; E2E denial tests. |
| AFF-DW-002 Contract-first API usage | Widgets call generated clients from approved OpenAPI contracts only; no invented endpoints. | OpenAPI diff; generated-client import checks; Pact/Spring Cloud Contract where applicable. |
| AFF-DW-003 MicroFunction binding | State-changing widget actions map to approved capability_code and microfunction_transaction_code. | Config schema validation; integration tests; policy decision record checks. |
| AFF-DW-004 Ports and adapters | Backend application/domain logic does not depend on framework, database, Kafka, OPA, Flowable, Keycloak, or AI provider SDKs directly. | ArchUnit/package rules; dependency graph; GitNexus drift findings. |
| AFF-DW-005 DDD boundary safety | Components, APIs, events, migrations, and MicroFunctions stay within owning bounded context unless contractually integrated. | Module boundary rules; schema ownership checks; contract registry evidence. |
| AFF-DW-006 Policy as code | RBAC/SBAC/ABAC decisions come from OPA/Rego or approved policy abstraction, not hidden UI/controller if-statements. | Semgrep rules; Rego unit tests; deny-by-default tests. |
| AFF-DW-007 Observability by design | Critical paths emit trace_id, span_id, evidence_ref, policy_decision_id, capability_code, and classification-safe logs/metrics. | OTel instrumentation tests; log redaction tests; dashboard evidence. |
| AFF-DW-008 Reversibility | Templates, layouts, config, toggles, components, policy bundles, and migrations have rollback, forward-fix, deactivation, or compensation path. | PR/MR evidence checklist; restore/replay tests; release-readiness gate. |

# 7. Security, OPA/SBAC, and Policy Testing
| Test class | Required scenarios | Evidence |
| --- | --- | --- |
| RBAC/SBAC/ABAC tests | Allowed role/skill/attribute combinations; denied unauthorized users; tenant/branch/classification ceiling; separation-of-duties conflicts. | OPA/Rego tests, policy bundle checksum, decision logs. |
| Fail-closed tests | OPA unavailable, session unavailable, stale token, missing classification, missing capability binding, missing audit/evidence writer. | Integration report proving deny/safe response. |
| Authenticated DAST | Synthetic authenticated user scans for workspace routes, admin builder, widget APIs, upload endpoints, AI panel APIs, and denial paths. | ZAP/authenticated DAST report with synthetic data and remediation evidence. |
| Secrets and token tests | No passwords, raw JWTs, tokens, secrets, raw PII, account numbers, KYC documents, or prompts in browser storage, logs, telemetry, screenshots, or evidence. | Secret scan, log redaction tests, browser storage assertion, screenshot review. |
| Unsafe rendering tests | No unsafe HTML by default, no template injection, no arbitrary component loading, no cross-site scripting path, safe markdown rendering. | SAST, component tests, DAST, secure renderer review. |
| Admin and toggle tests | Only authorized makers/checkers/approvers can change templates, policies, runtime toggles, and activation states. | Workflow approval record, OPA decision, audit event. |

# 8. Accessibility, UX, and Responsive Testing
| Area | Required proof |
| --- | --- |
| Keyboard and focus | All controls, menus, widgets, modals, AI panel, admin builder, and drag/drop alternatives are keyboard operable with visible focus and logical order. |
| Screen reader semantics | Labels, roles, names, descriptions, live regions, error associations, and landmark structures are tested for dynamic content and async state changes. |
| Contrast, motion, and readability | Approved contrast thresholds, reduced-motion preference, responsive text behavior, and safe error/empty/denied states are validated. |
| Responsive layout | Desktop, tablet, and approved mobile views are tested for layout priority, widget resizing rules, AI drawer behavior, and no clipped critical content. |
| User safety | Disabled and denied actions explain safe reasons without leaking policy internals; high-impact actions require confirmation and human approval where applicable. |

# 9. MicroFunction, API, Event, and Database Contract Testing
| Contract area | Mandatory tests |
| --- | --- |
| MicroFunction | Transaction code exists, step order respects receive/correlate/session/rate/authorize/validate/idempotency/execute/audit/outbox/observe/respond patterns, and failure/compensation paths are tested. |
| OpenAPI | Request/response schema, safe response envelope, Problem Details/error model, idempotency header, pagination, authorization, classification, and generated client compatibility are tested. |
| AsyncAPI / Kafka / Avro / CloudEvents | Event schema compatibility, ce-id/trace/correlation/causation fields, producer/consumer contract, schema evolution, retry topic, DLQ, and replay safety are tested. |
| Database / Flyway | Workspace configuration, component catalog, policy binding, runtime toggle, evidence profile, RLS, indexes, seed data, and rollback/forward-fix migrations pass clean-migrate and drift checks. |
| Outbox / Inbox / Idempotency | Duplicate action requests, browser retries, Kafka replays, callback retries, and DLQ replay do not duplicate business effects or bypass audit/evidence. |

# 10. Observability, Runtime Toggle, and Evidence Testing

Trace tests must correlate browser interaction, gateway request, OPA decision, backend API, MicroFunction step, database access, Kafka/outbox event, audit record, and evidence record using trace_id, request_id, correlation_id, causation_id, policy_decision_id, and evidence_ref.

Runtime telemetry toggles must be tested for default-safe behavior, authorized change path, audit record, propagation delay, rollback, sampling boundaries, and performance impact.

Forbidden telemetry tests must block passwords, tokens, raw JWTs, secrets, raw PII, account numbers, raw documents, raw KYC files, embeddings, private keys, unrestricted customer payloads, and high-cardinality identifiers as metric labels.

Dashboards must show workspace health, policy denials, cache hit/miss, slow widgets, widget action success/failure, AI guardrail outcomes, template lifecycle, MicroFunction traceability, and evidence completeness.
| Evidence artifact | Minimum content |
| --- | --- |
| workspace-test-evidence.json | workspace_code, screen_code, component_key, actor_hash, role/skill, classification, test_run_id, trace_id, evidence_ref. |
| policy-test-evidence.json | policy_ref, bundle_checksum, input fixture, allow/deny/require_approval result, reason_code, negative-test status. |
| accessibility-report.html/json | WCAG target, keyboard status, axe findings, manual review notes, exceptions, owner, remediation target. |
| runtime-toggle-report.json | toggle_code, old/new value, actor, approval, policy decision, audit event, propagation result, rollback test. |
| trace-reconstruction.md | User action to backend/event/audit/evidence chain with no sensitive payload exposure. |

# 11. Resilience, Heavy Transaction, and Cache-Loss Testing
| Scenario | Expected behavior |
| --- | --- |
| Redis/Valkey unavailable or stale | Workspace resolver falls back to authoritative PostgreSQL or returns safe degraded response; Redis is never treated as truth; cache invalidation evidence is produced. |
| OPA unavailable | Protected workspace, component, field, action, admin, and AI capabilities deny or require safe human-handled fallback. |
| Duplicate widget action | Idempotency key, inbox/outbox state, and MicroFunction execution records prevent duplicate business effects. |
| Kafka retry and DLQ replay | Consumer is idempotent, retry-safe, traceable, replayable, and evidence-producing; replay requires approved operator action. |
| Heavy personalization/update load | Concurrent layout saves, template activations, cache invalidations, and widget actions preserve consistency, locking, and safe response behavior. |
| Partial failure during action | Compensation, rollback, forward-fix, or safe deactivation path is executed or queued with evidence and owner assignment. |

## 15.2 Acceptance Criteria

All critical Dynamic Workspace flows have unit, component, integration, contract, E2E, policy, accessibility, security, observability, and evidence tests appropriate to risk.

Architecture fitness gates prevent frontend business authority, direct database calls, direct model calls, direct provider SDK use, MicroFunction bypass, and unregistered components.

OPA/SBAC/ABAC tests include allow, deny, require_approval, fail-closed, and separation-of-duties scenarios.

OpenAPI/AsyncAPI/Kafka/Avro/CloudEvents, outbox/inbox, idempotent consumers, DLQ/replay, and schema evolution tests are present where integration is affected.

Runtime telemetry toggles and feature flags are tested for authorization, audit, safe default, rollback, observability, and performance impact.

GitNexus and CI/CD evidence identify impacted components, APIs, policies, MicroFunctions, migrations, tests, and dashboards.

Auto-Heal, Auto-Learn, and Auto-Improve outputs remain candidate/proposal-only until evidence, tests, and human approvals are complete.

## 15.3 AVCI Compliance Summary
| AVCI property | Compliance evidence |
| --- | --- |
| Attributable | Defines owner, co-owners, RACI, test ownership, policy owner, component owner, source references, PR/MR actor, GitNexus commit SHA, and evidence_ref. |
| Verifiable | Requires executable tests, CI/CD gates, contract checks, OPA tests, accessibility tests, security scans, trace reconstruction, GitNexus evidence, and review approvals. |
| Classifiable | Requires classification on tests, fixtures, telemetry, screenshots, evidence, prompts, AI outputs, dashboards, and publication eligibility. |
| Improvable | Requires known limitations, recurring failure analysis, denial analytics, incident feedback, backlog items, and governed Auto-Heal/Auto-Learn/Auto-Improve loops. |

Final Adoption Rule: A Dynamic Workspace is accepted only when it is test-proven, policy-proven, evidence-proven, observable, accessible, reversible, secure, and AVCI-compliant.
